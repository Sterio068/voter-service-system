# 選民服務系統 v1.0.31 外部版反推需求說明書

文件版本：v1.0.31  
文件類型：由既有系統反推之 Product Requirements Document  
適用對象：客戶、驗收人員、外部審計者、外部開發團隊、AI 交接代理  
揭露範圍：本文件不包含本機資料路徑、內部原始碼位置、token、金鑰、部署機密或私有服務設定值。

---

## 1. 文件目的

本文件依據目前已完成之選民服務系統 v1.0.31 反推出產品需求，用於描述系統「應該滿足什麼需求」而非「程式如何實作」。文件可作為以下用途：

| 用途 | 說明 |
|---|---|
| 客戶驗收 | 以需求與驗收條件確認系統功能是否符合交付範圍。 |
| 外部審計 | 讓審計者理解功能邊界、資料流、權限與使用情境。 |
| 後續開發 | 作為新功能、重構、測試與估工的需求基準。 |
| AI 接手 | 讓外部 AI 能以產品需求角度理解整套系統。 |

---

## 2. 產品背景

民意代表服務處日常需要處理大量民眾服務工作，包含選民資料維護、陳情案件、地方團體互動、公文收發、行程安排、活動辦理、問卷回饋、禮儀支出與工作報表。若資料散落在 Excel、通訊軟體、紙本與個人電腦中，容易造成以下問題：

| 問題 | 影響 |
|---|---|
| 選民資料分散 | 不易查詢歷史互動、支持度、關係與服務紀錄。 |
| 案件追蹤困難 | 陳情進度、逾期案件、承辦人責任難以掌握。 |
| 交接成本高 | 助理離職或換手時，歷史紀錄不完整。 |
| 報表需手工整理 | 每週、每月成果統計耗時且容易出錯。 |
| 權限與稽核不足 | 難以追蹤誰匯出、修改或刪除重要資料。 |
| 部署門檻 | 服務處通常不具備維護大型雲端系統的能力。 |

因此，系統需求是提供一套可本地安裝、可小團隊共用、功能完整、具備角色權限與資料追蹤能力的服務處管理系統。

## 3. 產品目標與非目標

### 3.1 產品目標

| 編號 | 目標 | 成功判準 |
|---|---|---|
| G-01 | 集中管理選民與服務資料 | 使用者可由單一系統查詢選民、案件、行程、團體與互動紀錄。 |
| G-02 | 標準化陳情處理流程 | 每件陳情具備案號、狀態、承辦人、期限與處理紀錄。 |
| G-03 | 支援服務處團隊協作 | 不同角色可依權限新增、查看、編輯、刪除或管理資料。 |
| G-04 | 降低部署與維護門檻 | 系統可以桌面安裝檔交付，不需客戶自行架設複雜雲端服務。 |
| G-05 | 提供統計與管理視角 | 管理者可透過儀表板與報表查看案件、地區、活動與團隊效率。 |
| G-06 | 支援資料交換 | 系統可匯入、匯出與列印常用業務資料。 |
| G-07 | 支援交付後更新 | 系統可檢查版本更新，並支援受控下載與安裝。 |

### 3.2 非目標

| 編號 | 非目標 | 說明 |
|---|---|---|
| NG-01 | 不作為大型雲端 SaaS 多租戶平台 | 系統定位為本地端或小型團隊使用。 |
| NG-02 | 不取代正式政府公文系統 | 公文模組用於服務處內部管理與列印，不等同機關正式電子公文交換。 |
| NG-03 | 不保證大規模高併發寫入 | 資料庫架構適合服務處規模，不適合大量使用者同時高頻寫入。 |
| NG-04 | 不內建所有法遵流程 | 系統提供工具，正式個資治理仍需組織制度配合。 |
| NG-05 | 不強制依賴外部 AI 或雲端服務 | AI、Google Calendar、LINE 等為可選整合。 |

## 4. 使用者與角色需求

### 4.1 使用者角色

| 角色 | 描述 | 核心需求 |
|---|---|---|
| 系統管理者 | 負責帳號、權限、設定、備份、版本與稽核 | 需要完整管理權限與系統健康資訊。 |
| 服務處主任 | 負責督導案件、助理與工作進度 | 需要儀表板、報表、承辦人負荷與逾期提醒。 |
| 一般助理 | 負責每日接案、登錄選民、處理公文與行程 | 需要快速搜尋、新增、追蹤與列印。 |
| 志工 | 協助活動、查詢資料或簡單服務 | 需要有限查看權限，避免誤改資料。 |
| 議員或主管 | 需要掌握整體民意、服務成果與地方重點 | 需要報表、趨勢、今日行程與高風險案件。 |
| 外部驗收者 | 驗證系統是否符合交付需求 | 需要明確功能清單與驗收標準。 |

### 4.2 高層使用者故事

| 編號 | 使用者故事 |
|---|---|
| US-01 | 身為助理，我希望能快速查詢選民資料，以便接電話時立即知道過往服務紀錄。 |
| US-02 | 身為助理，我希望能建立陳情案件並記錄每次處理，以便追蹤進度。 |
| US-03 | 身為主任，我希望能查看逾期案件與承辦人負荷，以便安排工作。 |
| US-04 | 身為管理者，我希望能設定帳號權限，以便控制不同人員可操作的功能。 |
| US-05 | 身為管理者，我希望系統能定期備份，以便資料出問題時可還原。 |
| US-06 | 身為助理，我希望能匯入 Excel 選民資料，以便快速建立初始資料。 |
| US-07 | 身為主管，我希望能看到地區、案件類型、活動成效與滿意度報表，以便規劃服務方向。 |
| US-08 | 身為使用者，我希望下拉選單缺少類別時能快速前往類別設定，以便不中斷資料建立流程。 |

## 5. 系統範圍

### 5.1 需求範圍內

| 範圍 | 包含內容 |
|---|---|
| 身分與權限 | 登入、登出、目前使用者、改密碼、角色權限。 |
| 選民管理 | 選民 CRUD、搜尋、篩選、標籤、關係、互動、合併、匿名化、匯入匯出、列印。 |
| 陳情管理 | 案件 CRUD、案號、狀態、承辦、期限、處理紀錄、統計、匯入匯出。 |
| 任務管理 | 任務 CRUD、今日任務、批次指派、批次完成、批次刪除。 |
| 公文管理 | 收發文、文號、狀態、查詢、列印、附件。 |
| 行程與諮詢 | 月曆、跨日行程、衝突檢查、諮詢時段與預約。 |
| 團體管理 | 團體 CRUD、成員、角色、相關行程與支出。 |
| 活動與問卷 | 活動 CRUD、參與者、問卷、題目、回應、統計、轉陳情。 |
| 通知管理 | 通知草稿、編輯、刪除、送出與觸及報表。 |
| 禮儀與支出 | 禮儀紀錄、禮品類別、廠商、支出摘要、預算。 |
| 提案追蹤 | 提案 CRUD、狀態、統計、匯出。 |
| 報表 | 地區、案件、團隊、活動、滿意度、問卷與通知等報表。 |
| 系統後台 | 使用者、類別、設定、備份、還原、稽核、健康、資料品質、資料保留。 |
| 整合 | AI、Google Calendar、LINE、更新檢查。 |

### 5.2 需求範圍外

| 範圍外項目 | 說明 |
|---|---|
| 多租戶雲端帳務 | 不提供多客戶租戶、線上訂閱、帳單管理。 |
| 大規模雲端分散式資料庫 | 不設計為數百人同時高併發使用。 |
| 官方電子公文交換 | 不與政府正式電子公文交換標準直接串接。 |
| 公開民眾入口網站 | 目前以服務處內部作業為主。 |
| 完整 CRM 行銷自動化 | 通知與活動管理存在，但不是大型行銷自動化平台。 |

## 7. 認證與權限需求

### FR-AUTH-01 登入

| 欄位 | 說明 |
|---|---|
| 優先級 | 必須 |
| 需求 | 系統必須提供帳號密碼登入。 |
| 驗收條件 | 使用正確帳密可登入；錯誤帳密不可登入；登入後載入使用者名稱與角色。 |

### FR-AUTH-02 登出

| 欄位 | 說明 |
|---|---|
| 優先級 | 必須 |
| 需求 | 使用者必須可主動登出。 |
| 驗收條件 | 登出後不可存取需登入頁面；重新進入需再次登入。 |

### FR-AUTH-03 目前使用者

| 欄位 | 說明 |
|---|---|
| 優先級 | 必須 |
| 需求 | 系統必須能取得目前登入者資訊，且不得顯示密碼雜湊。 |
| 驗收條件 | 前端可顯示使用者名稱與角色；回應內容不包含密碼。 |

### FR-AUTH-04 修改密碼

| 欄位 | 說明 |
|---|---|
| 優先級 | 必須 |
| 需求 | 使用者必須可修改自己的密碼。 |
| 驗收條件 | 密碼修改成功後可用新密碼登入；舊密碼失效。 |

### FR-AUTH-05 角色權限

| 欄位 | 說明 |
|---|---|
| 優先級 | 必須 |
| 需求 | 系統必須依角色限制模組與操作。 |
| 驗收條件 | 志工不可新增或刪除選民；非管理者不可進入使用者管理；權限不足時顯示明確提示。 |

## 9. 選民管理需求

### FR-VOTER-01 選民列表

| 欄位 | 說明 |
|---|---|
| 優先級 | 必須 |
| 需求 | 系統必須提供選民列表，支援分頁、搜尋、縣市、鄉鎮區、村里、標籤與狀態篩選。 |
| 驗收條件 | 新增選民後可在列表查到；搜尋姓名或電話可返回正確結果；軟刪除資料預設不顯示。 |

### FR-VOTER-02 新增選民

| 欄位 | 說明 |
|---|---|
| 優先級 | 必須 |
| 需求 | 使用者必須可新增選民基本資料、聯絡資料、地址、標籤與備註。 |
| 驗收條件 | 姓名必填；成功建立後可進入詳情頁；新增操作進入稽核紀錄。 |

### FR-VOTER-03 編輯選民

| 欄位 | 說明 |
|---|---|
| 優先級 | 必須 |
| 需求 | 授權使用者必須可修改選民資料。 |
| 驗收條件 | 修改手機、地址、標籤後詳情頁與列表同步更新；未授權角色不可修改。 |

### FR-VOTER-04 選民詳情

| 欄位 | 說明 |
|---|---|
| 優先級 | 必須 |
| 需求 | 系統必須提供選民詳情頁，整合基本資料、標籤、關係人、關注議題、互動程度、聯絡紀錄、活動歷史與附件。 |
| 驗收條件 | 從列表點選選民可進入詳情；各分頁資料可載入；無資料分頁顯示空狀態。 |

### FR-VOTER-05 標籤管理

| 欄位 | 說明 |
|---|---|
| 優先級 | 必須 |
| 需求 | 系統必須支援選民多標籤，例如樁腳、志工、支持者、捐款者、意見領袖或自訂標籤。 |
| 驗收條件 | 同一選民可設定多個標籤；標籤可作為列表篩選條件。 |

### FR-VOTER-06 人際關係

| 欄位 | 說明 |
|---|---|
| 優先級 | 應有 |
| 需求 | 系統應支援建立選民間關係，例如配偶、親屬、鄰居、朋友、同事、樁腳等。 |
| 驗收條件 | 可新增與刪除關係；關係資料顯示於選民詳情。 |

### FR-VOTER-07 關注議題與互動程度

| 欄位 | 說明 |
|---|---|
| 優先級 | 應有 |
| 需求 | 系統應可記錄選民關注議題、支持度、是否為關鍵支持者與志工狀態。 |
| 驗收條件 | 編輯後資料可保存；報表可使用相關資料分析。 |

### FR-VOTER-08 聯絡紀錄與活動歷史

| 欄位 | 說明 |
|---|---|
| 優先級 | 必須 |
| 需求 | 系統必須支援記錄與選民的聯絡紀錄，並整合陳情、活動、問卷等歷史事件。 |
| 驗收條件 | 新增聯絡紀錄後可在選民詳情看到；活動歷史依時間排序。 |

### FR-VOTER-09 重複檢查

| 欄位 | 說明 |
|---|---|
| 優先級 | 應有 |
| 需求 | 系統應可依手機或身分識別欄位找出疑似重複選民。 |
| 驗收條件 | 建立兩筆相同手機測試資料後，重複檢查能列出疑似重複。 |

### FR-VOTER-10 選民合併

| 欄位 | 說明 |
|---|---|
| 優先級 | 應有 |
| 需求 | 系統應允許授權使用者將重複選民合併，並轉移相關歷史資料。 |
| 驗收條件 | 合併後舊選民不在一般列表顯示；陳情、聯絡紀錄與活動歷史保留於目標選民。 |

### FR-VOTER-11 軟刪除與匿名化

| 欄位 | 說明 |
|---|---|

| 驗收條件 | 軟刪除後預設列表不顯示；匿名化後主要個資欄位不再顯示原值；操作進入稽核紀錄。 |

|---|---|

## 23. 核心業務規則

| 編號 | 規則 |
|---|---|
| BR-01 | 使用者必須登入後才能操作受保護資料。 |

| BR-03 | 選民刪除採軟刪除，預設列表不顯示。 |

| BR-08 | 公文文號依年度與收發文類型流水產生，不可重複。 |
| BR-09 | 行程新增與更新需檢查時間重疊。 |
| BR-10 | 跨日行程需在涵蓋日期區間內被查詢到。 |
| BR-11 | 法律諮詢預約不可超過時段容量。 |
| BR-12 | 備份還原前必須檢查備份有效性。 |
| BR-13 | 高風險操作需記錄稽核資料。 |
| BR-14 | 類別型欄位應可由管理者維護。 |
| BR-15 | 前端無資料、錯誤與載入狀態需明確呈現。 |

## 28. 結論

依目前 v1.0.31 系統能力反推，選民服務系統的主要需求是建立一套本地端、小團隊可用、功能完整且可驗收的服務處管理平台。它的核心不只是保存選民名冊，而是把選民、陳情、公文、行程、團體、活動、問卷、通知、禮儀支出、提案、報表與後台治理整合成一套可日常運作的工作系統。

本需求說明書可作為交付驗收、外部審計、後續開發與 AI 接手的基準文件。後續若新增功能，建議依本文件格式補充需求編號、使用者故事、業務規則與驗收條件，避免功能完成但驗收標準不明。
