# Data Quality Runbook

## 目標

將重複選民、孤兒附件、壞關聯資料與缺漏欄位修復成可追蹤、可回復、可稽核的流程。

## 例行掃描

1. 進入「系統設定 → 資料品質掃描」。
2. 先檢查高風險項目：重複身分證、重複手機、孤兒附件、附件檔案遺失。
3. 匯出或截圖保存掃描結果，並記錄掃描時間與操作者。
4. 修復前先建立手動備份，並確認備份顯示「已簽章」。

## 重複選民處理

1. 以身分證、手機、姓名與地址交叉比對，不只依單一欄位判斷。
2. 保留資料較完整、關聯資料較多的一筆作為主檔。
3. 合併前檢查 petitions、contact_records、tasks、events、survey_responses、attachments 是否會改指向主檔。
4. 合併後抽查選民詳情頁、陳情列表、聯絡紀錄與 audit logs。

## 孤兒附件處理

1. 確認 `ref_type` 與 `ref_id` 對應主檔是否存在。
2. 若主檔被誤刪，先還原主檔或重新連結附件。
3. 若附件確定無合法歸屬，先備份檔案，再從資料庫刪除附件紀錄。
4. 遺失檔案不可直接重建空檔；需標記為缺失並通知資料負責人補件。

## 壞關聯資料處理

1. 關聯表修復前先列出受影響 ID 與目標表。
2. 優先使用既有 API 或後台操作，避免手改 SQLite。
3. 必須手動 SQL 時，先在複本 DB 驗證查詢與 rollback 策略。
4. 修復完成後跑 `npm run verify`，並重新執行資料品質掃描。

## 稽核要求

- 合併、匿名化、刪除、完整匯出、備份還原都必須有 audit log。
- 手動 SQL 修復需補一筆人工維運紀錄，包含原因、SQL 摘要、備份檔名、驗證結果。
- 涉及個資外洩疑慮時，改依 `DATA_PROTECTION.md` 與 `INCIDENT_RESPONSE.md` 處理。
